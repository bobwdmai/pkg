# AI OS (Aider-Inspired Desktop Agent)

AI OS is a local, offline-first coding assistant with:

- Chat interface (Aider-style prompt loop)
- Code editor + run console
- Ollama model switching:
  - local fast model (`qwen2.5-coder:3b`)
  - local fallback model (`qwen2.5-coder:1.5b`)
  - remote heavy model host (`10.0.0.133`)
- `.deb` packaging for Linux desktop install

## Model Architecture Summary (Codex Version)

AI OS uses a multi-model Ollama setup:
- `qwen2.5-coder:1.5b` for prompt optimization and lightweight/fallback reasoning
- `qwen2.5-coder:3b` as the default local coding assistant
- `qwen2.5-coder:14b` on remote host `10.0.0.133:11434` for complex reasoning

### Models used

#### `qwen2.5-coder:1.5b` (lightweight / prompt optimizer)
- Purpose:
  - fast prompt rewriting
  - lightweight reasoning
  - fallback path when local resources are constrained
- Used for:
  - optional prompt optimization before main inference
  - fast transformation of user input into structured coding prompts

#### `qwen2.5-coder:3b` (main local coding assistant)
- Purpose:
  - primary local AI model
  - code generation, explanation, and debugging
  - default chat model in AI OS local mode
- Optimized for:
  - code reasoning
  - instruction following
  - local CPU/GPU balance

#### `qwen2.5-coder:14b` (remote server brain)
- Endpoint: `http://10.0.0.133:11434`
- Purpose:
  - heavy reasoning
  - complex code generation and debugging
  - high-intelligence server mode for advanced tasks
- Used when:
  - task complexity exceeds local 3B capability
  - AI OS switches to server mode

### Multi-model routing

| Task type | Model used |
| --- | --- |
| simple chat | `qwen2.5-coder:1.5b` |
| coding / UI / scripts | `qwen2.5-coder:3b` |
| complex debugging / architecture | `qwen2.5-coder:14b` (remote) |

### AI pipeline (Aider-style)

1. User submits prompt in the GUI.
2. Optional prompt optimization step runs on `qwen2.5-coder:1.5b`.
3. Main response is generated by `qwen2.5-coder:3b` or `qwen2.5-coder:14b`.
4. Response is rendered in a ChatGPT-style UI.
5. Optional post-steps:
   - execute code
   - show output
   - speak response via TTS

### Agent-style behavior (early version)

Early agent experiments included:
- prompt rewriting agent
- execution agent (Python subprocess run path)
- dynamic multi-model switching

This follows an Aider-like loop:
- read prompt -> generate code -> optionally execute -> return result

### Extra system features

- Text-to-speech with `espeak-ng` for spoken responses
- Code execution engine with temporary Python file execution
- Sandboxed subprocess runner for controlled runs

### One-line architecture summary

AI OS is a local multi-model coding assistant + IDE + agent system that routes prompts across 1.5B/3B/14B Qwen models, optionally rewrites prompts, executes generated code when needed, and returns results in a ChatGPT-style desktop UI with optional TTS.

```bash
python3 -m ai_os.app
```

## Build `.deb`

Requirements:
- `dpkg-deb`
- `bash`

Build:

```bash
bash scripts/build_deb.sh
```

Output package goes to:

```text
dist/ai-os_<version>_all.deb
```

## Runtime notes

- By default, local Ollama endpoint is `http://127.0.0.1:11434`.
- Remote heavy model endpoint defaults to `http://10.0.0.133:11434`.
- The app writes executable snippets to `/tmp/ai_os_runs`.
- The Run button executes Python code from the editor.

## Why this avoids past breakages

- Explicit Python package structure with `__init__.py`
- No dependency on global `pip` installs
- Launcher exports a deterministic `PYTHONPATH=/opt/ai-os`
- No virtualenv required inside target host
